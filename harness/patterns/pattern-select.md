# pattern-select — 가장 단순한 해부터

화려한 멀티에이전트 오케스트레이션은 **비용·디버깅난도·비결정성**을 동시에 올린다. 황금률: **될 만큼만 복잡하게.** 한 단계 올리기 전에 "더 단순한 게 정말 안 됐나"를 증명해야 한다. 모델 배분은 [[model-tiering]], 품질루프는 [[evaluator-optimizer]].

## 결정트리 (위에서부터, 막히면 한 칸 내려간다)
```
0. 한 번의 단일 프롬프트(+few-shot)로 되나?           → 그걸로 끝. 멈춰라.
1. 컨텍스트/도구가 필요? (검색·계산·외부API)          → 단일 에이전트 + JIT 도구루프(depth≤3)
2. 산출 품질이 부족? (채점에서 떨어짐)                 → evaluator-optimizer 루프(생성→judge→수정)
3. 진짜 독립·병렬 서브태스크가 다수 + 컨텍스트 격리?  → 멀티에이전트(worktree-per-agent)
4. 런타임 이상수복(봇/배포)?                          → OODA self-heal (빌드패턴 아님)
```
- 0→1→2 까지가 대부분의 일. **3(멀티에이전트)은 예외**여야 한다.
- 빌드/기능 개선 = PDCA 계열, 시간민감 수복 = OODA 계열 — 둘을 섞지 마라([[evaluator-optimizer]] 루프는 전자).

## 오케스트레이션 선택 (어느 스킬·사단을 언제 — '복잡도 한 칸'의 실체)
한 단계 올릴 때 대부분 **기존 스킬/루틴 호출**로 끝난다(재구현 금지 — 호출계약은 [[feedback-loop]]·*당신 환경의 것을 연결*).
| 상황 | 연결할 것(예) |
|---|---|
| 단일 산출 품질부족 | 당신의 심층-피드백/적대비평 스킬 (전수조사→실증→무한 90점 게이트) |
| 같은 문제 다관점 교차검증·의견충돌 | 당신의 다관점 토론 스킬 (3자 교차검증) |
| 신규 앱/주요기능 풀스택(다컴포넌트) | 당신의 멀티에이전트 빌드 스킬/스쿼드 (N 페르소나·무한 PM 게이트) |
| 런타임 봇/배포 이상 | 당신의 런타임 자가수복 루틴 ≈ [[self-healing]] |
- 셋 다 **단일 baseline → 실패지점 제시 후에만**. 스킬의 "무제한 반복"도 harness에선 cap6 ∧ P0/P1=0 ∧ HITL로 bounded.

## ★ 멀티에이전트 전에 단일 baseline 먼저
멀티에이전트로 바로 가면 **개선폭을 측정할 기준선이 없다.** 단일 에이전트 baseline을 먼저 세우고, 그게 실패하는 **구체 지점**을 보여야 멀티가 정당화된다.
- 보고된 추정으로 **대부분(~64%) 케이스는 단일 에이전트로 충분**하다 — 멀티는 토큰 N배·디버깅 N배인데 품질이득은 종종 0. (외부 연구 보고치 — 당신의 직접 측정이 아니니 baseline으로 검증하라.)
- "독립·병렬"이 진짜인지 의심하라: 서브태스크가 서로의 출력을 기다리면(순차 의존) 그건 멀티가 아니라 단일 파이프라인이다. fan-out은 **상호 무관한** 작업에만.
- 격리의 진짜 이유는 병렬성보다 **컨텍스트 오염 방지**: 검색 디테일을 sub-agent에 가두고 1~2k 토큰 요약만 회수(부모 컨텍스트 보호).

## model-tiering (붙은 패턴 — 상세 [[model-tiering]])
복잡도를 올릴 때 **모든 노드에 최상위 모델을 쓰지 마라.** 역할별 티어:
| 역할 | 모델 티어 | 이유 |
|---|---|---|
| orchestrator / 계획 | Opus(상위) | 분기·종합·판단 |
| worker / 대량 단순 | Haiku(하위) | 검색·추출·1차 스크리닝 = 싸게 |
| 최종 judge | **다른 패밀리** | self-preference 회피([[rubric]]) |
- 동반 가드: 동시성 캡 + 비용 원장 + $ 킬스위치(→ [[guardrails]]).

## 흔한 오용
- **패턴을 위한 패턴**: 멀티에이전트가 멋져서 쓰는 것 — baseline 없이는 금지.
- **0단계 건너뛰기**: 단일 프롬프트로 될 일에 도구루프/루프채점을 두름(토큰낭비).
- **티어 단일화**: 전 노드 Opus = 비용폭주. 전 노드 Haiku = 품질붕괴. 역할별 배분이 정답.

## 결정 / 규칙
- 한 단계 복잡도를 올릴 때마다 **"더 단순한 게 왜 안 됐나"를 한 줄로 적어라.** 못 적으면 내려가라.
- 멀티에이전트는 단일 baseline의 실패지점을 제시한 뒤에만.
- 모델은 역할별 티어링, 최종 judge는 항상 타 패밀리.
